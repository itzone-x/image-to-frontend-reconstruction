# 我做了个“设计稿翻译器”：专治 AI 原型落地时的走样

最近我遇到一个很现实的问题。

Stitch、Figma、GPT Image 这类工具，生成产品设计稿是真的漂亮。按钮有层次，卡片有呼吸感，配色也挺像那么回事。乍一看，仿佛产品已经快上线了。

然后你准备把它做成页面。

好，问题来了：我不会前端。

再往下，还有第二个问题：我身边也没有一个专业 UI 设计师，坐在旁边帮我把这张图一块一块切出来。

于是事情就变成了这样：设计稿在屏幕里像精装样板间，到了代码里，一不小心就变成了“同户型毛坯房”。结构大概还在，气质已经离家出走。

我当然也试过让最强的大模型直接照着图片写页面。GPT 5.5、Claude Code Opus 4.7 这种级别，已经很厉害了，但它们有个天然问题：它们是在“看图画页面”，不是在“切图做页面”。

这两件事听起来差不多，实际差得很远。

看图画页面，就像你让一个人看着招牌菜复刻一盘。他大概知道这盘菜是红烧的，也知道上面撒了葱花，但葱花撒在哪、酱汁怎么挂、盘边那点小装饰是什么味儿，就全靠猜。

前端还原也是一样。模型能看出“这里有个绿色图标”“背景有一点线条”“按钮旁边有个小星星”，然后它会努力用 CSS 或 SVG 画一个“差不多”的。第一眼还行，第二眼就开始不对劲：图标不像，间距飘了，按钮文字换行了，原本高级的背景纹理变成了手绘简笔画。

所以我做了一个 skill，名字有点长，叫 `image-to-frontend-reconstruction`。

它不是一个神奇按钮，也不是那种“上传图片，三秒生成完美页面”的营销魔法。它更像一套干活规矩：先拆，再切，再写。

简单说，它会把一张设计稿拆成三层。

第一层是前端原生内容，比如文字、按钮、输入框、卡片布局。这些不能直接糊进图片里，否则后面改文案、做适配、加交互都会很痛苦。

第二层是必须从原图里切出来的东西，比如复杂图标、背景纹理、装饰线条、插画碎片。这些东西让模型硬画，很容易画歪。既然原图里已经有了，就别逞强，老老实实裁。

第三层才是可以用代码画的简单元素，比如边框、阴影、圆角、分割线。低风险的东西交给 CSS，高细节的东西交给切图。

听起来不复杂，对吧？确实不复杂。难的是每次都按这个流程来，不要一上来就“让 AI 自由发挥”。

我自己拿它试了不少场景。To C 产品的高保真原型，大概能还原到 90% 左右。有些后台系统页面，因为组件更规整，甚至能接近原样复刻。这里面没有玄学，主要就是少让模型猜，多让它从原图拿证据。

这个 skill 真正解决的，是一个夹在设计和开发之间的尴尬问题：

你有一张漂亮的设计稿，但你没有设计师切图，也不想让大模型把它“画成另一张图”。

它适合像我这样的人：懂一点产品，想把想法快速落地，但不想每次都卡在“为什么这个页面看起来就是差一口气”上。

当然，它也有边界。字体渲染、不同设备宽度、平台控件差异，还是需要人看一眼。它不能替你审美，也不能替你上线背锅。这个锅，暂时还是人类专属。

但至少，它能把那种“明明设计稿很美，做出来却像远房亲戚”的概率降下来。

对我来说，这就够实用了。

毕竟很多时候，我们差的不是一个会吟诗的 AI，而是一个愿意认真切图、认真量间距、认真把按钮文字别挤换行的小工友。
